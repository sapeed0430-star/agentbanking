01 / Proof flow

A four-step story from policy gate to signed receipt.

The page reads like a system diagram but keeps the prose human, so both product and ops can follow the flow quickly.

1

Policy gate

Validate the request against the operating rules before any receipt is emitted.

2

Evidence capture

Attach inputs, digests, and supporting references while the context is still fresh.

3

Verification run

Confirm signatures, timestamp anchors, and transparency proofs in one place.

4

Signed receipt

Return a compact artifact that can be replayed later without a live dependency.

02 / Capability Cards

Three surfaces that turn proof into something readable.

Each card is intentionally dense enough to feel operational, while still keeping the main page elegant and approachable.

Receipts

Receipts are legible and exportable.

Make the receipt the source of truth for each verification, with clear digest and status metadata.

Includes receipt ID, evidence references, and outcome state.
Offline

Verification can be replayed without the live service.

Use the same proof layout to explain what was checked even when the network is unavailable.

Optimized for incident review and field debugging.
Audit

The UI reads like an operations tool, not a brochure.

Surface health, trust material, and failure mode cues in the same visual language as the product.

One page for engineering, compliance, and leadership.
03 / Operations

Keep the page useful when the system is under stress.

The lower half of the page shifts from marketing to status and design intent so it can support implementation and review.

System posture

  • Signer mode: local-ed25519
  • Timestamp mode: rfc3161
  • Transparency mode: rekor
  • Fallback state: ready

Design notes

  • Use one accent color per semantic purpose.
  • Keep the proof panel elevated above the hero surface.
  • Stack the mobile flow so CTA, proof panel, and stats remain visible in order.
  • Respect reduced-motion preferences by removing float and shimmer effects.
